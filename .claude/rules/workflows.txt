# Agentic Workflows

Agentic workflows allow microservices to collaborate on multi-step processes. A workflow is a directed graph of tasks orchestrated by the Foreman core service. Each task is a standalone endpoint that reads from and writes to a shared state carried by a `*workflow.Flow`.

## Task Endpoints

A task endpoint is a functional endpoint that performs a single step of a workflow. Its signature always includes `ctx context.Context` and `flow *workflow.Flow` as the first two parameters, followed by input arguments sourced from the flow's state, and returns output arguments written back to the state plus `err error`.

```go
import "github.com/microbus-io/fabric/workflow"

func (svc *Service) VerifyCredit(ctx context.Context, flow *workflow.Flow, creditScore int) (creditVerified bool, err error) {
	if creditScore >= 700 {
		return true, nil
	}
	return false, nil
}
```

Input arguments are matched by name to fields in the flow's state. Output arguments are written back to the state under their name. An output argument with an `Out` suffix (e.g. `creditScoreOut`) implements a read-modify-write pattern - it appears as both input and output.

**Prefer typed arguments over `flow.Get`/`flow.Set`.** Task inputs and outputs should be expressed as typed function arguments wherever the types are available in the API package. This makes the task's state contract explicit in the signature and lets the marshaler handle serialization. Reserve `flow.Get`/`flow.Set` only for internal types that are not part of the public API (e.g. intermediate state shared between tasks within the same microservice).

```go
// Typed arguments make the state contract explicit
func (svc *Service) ProcessResponse(ctx context.Context, flow *workflow.Flow, llmContent string, toolRounds int) (toolsRequested bool, toolRoundsOut int, err error) {
    // llmContent and toolRounds are read from state automatically
    // toolsRequested and toolRoundsOut are written back to state automatically
}
```

Tasks are registered on port `:428` by default.

## Workflow Graph Endpoints

A workflow graph endpoint returns a `*workflow.Graph` that defines the structure of a workflow - its tasks, transitions, and merge strategies. The graph is fetched once when a flow is created and stored with it.

```go
import "github.com/microbus-io/fabric/workflow"

func (svc *Service) CreditApproval(ctx context.Context) (graph *workflow.Graph, err error) {
	submitApp := myserviceapi.SubmitApplication.URL()
	verifyCredit := myserviceapi.VerifyCredit.URL()
	verifyEmployment := myserviceapi.VerifyEmployment.URL()
	finalDecision := myserviceapi.FinalDecision.URL()

	graph = workflow.NewGraph(myserviceapi.CreditApproval.URL())

	// Fan-out: submitApp has two normal outgoing transitions, so verifyCredit and verifyEmployment run as parallel branches
	graph.AddTransition(submitApp, verifyCredit)
	graph.AddTransition(submitApp, verifyEmployment)
	// Fan-in: every parallel branch must converge on a node marked with SetFanIn, or graph.Validate rejects the graph
	graph.SetFanIn(finalDecision)
	graph.AddTransition(verifyCredit, finalDecision)
	graph.AddTransition(verifyEmployment, finalDecision)
	// Conditional transition
	graph.AddTransitionWhen(finalDecision, workflow.END, "creditVerified")
	// Goto transition (taken only when task calls flow.Goto)
	graph.AddTransitionGoto(finalDecision, submitApp)

	return graph, nil
}
```

Task URLs in the graph are obtained from the generated API package constants (e.g. `myserviceapi.TaskName.URL()`).

## The `workflow` Package

**`workflow.Flow`** is the carrier passed to every task. It provides typed accessors for reading and writing state:

- Read: `GetString`, `GetInt`, `GetFloat`, `GetBool`, `GetDuration`, `GetStrings`, `Has`, `Get` (for complex types), `ParseState` (unmarshal into struct)
- Write: `SetString`, `SetInt`, `SetFloat`, `SetBool`, `SetDuration`, `SetStrings`, `Set` (for complex types)
- Struct-based: `Snapshot` captures state before modifications, `SetChanges(source, snap)` records only fields that differ from the snapshot

Control signals on `Flow`:

- `flow.Goto(taskURL)` - override transition routing to jump to a specific task
- `flow.Retry(maxAttempts, initialDelay, multiplier, maxDelay) bool` - re-execute this task with exponential backoff; returns true if attempts remain, false if exhausted
- `flow.RetryNow() bool` - re-execute this task immediately with no limit; shorthand for `Retry(math.MaxInt32, 0, 0, 0)`
- `flow.RetryOnTimeout(err, maxAttempts, initialDelay, multiplier, maxDelay) bool` - retry only when `err` carries HTTP status 408; returns false (so the caller can surface the error) for any other status
- `flow.Sleep(duration)` - delay the next step by a duration
- `flow.Interrupt(payload)` - pause the flow for external input (human-in-the-loop)
- `flow.Subgraph(workflowURL, input)` - dynamically launch a child workflow; the step is parked until the child completes, then re-executed with the child's output merged into state (see [Dynamic Subgraphs](#dynamic-subgraphs))

**`workflow.Graph`** defines workflow structure:

- `NewGraph(name)` - create a new graph
- `AddTask(taskURL)` - register a task node (first added becomes default entry point)
- `AddSubgraph(workflowName)` - register a child workflow as a subgraph node
- `SetEntryPoint(taskURL)` - override the default entry point
- `AddTransition(from, to)` - unconditional transition
- `AddTransitionWhen(from, to, when)` - conditional transition based on state expression
- `AddTransitionGoto(from, to)` - transition taken only when `flow.Goto` is called
- `AddTransitionForEach(from, to, forEach, as)` - dynamic fan-out over a state array field
- `SetFanIn(taskURL)` - mark a node as the fan-in nexus where parallel branches converge. Mandatory whenever the graph fans out: every fan-out source (a node with 2+ normal outgoing transitions, or any `forEach` transition) must converge on a `SetFanIn` node, otherwise `Validate` rejects the graph (see [Fan-Out and Fan-In](#fan-out-and-fan-in))
- `AddTransitionOnError(from, to)` - transition taken when the source task returns an error (see [Error Transitions](#error-transitions))
- `AddTransitionOnTimeout(from, to)` - error transition taken only when the source task's error carries HTTP status 408 (both foreman-side and task-side timeouts). Wins over a plain OnError when the status matches
- `SetReducer(field, reducer)` - escape hatch to override the prefix-driven reducer for a state field (rarely needed; see Reducers below)
- `DeclareInputs(fields...)` - declare which state fields are passed into this graph; no args = nothing, `"*"` = everything, named = those only
- `DeclareOutputs(fields...)` - declare which state fields are returned on completion; no args = nothing, `"*"` = everything, named = those only
- `workflow.END` - pseudo-node indicating workflow termination

**Reducers** control how state fields are merged at fan-in points. The reducer is normally chosen by the field name's prefix, with no explicit configuration:

- `sum*` - numeric add (e.g. `sumFailures`, `sumScore`)
- `list*` - array append, duplicates kept, ordered (e.g. `listMessages`, `listEvents`)
- `set*` - polymorphic: array union (dedupe) or object merge (new key wins) depending on the value's JSON kind (e.g. `setUsers`, `setRoles`)
- anything else - replace (last write wins)

The character right after the prefix must be uppercase. `summary`, `listening`, `setup` do not match; `sumScore`, `listMessages`, `setUsers` do. This boundary rule prevents accidental matches against English words.

**These prefixes are reserved. Do not name a state field or task argument `sum*`, `list*`, or `set*` unless it is meant to be reduced at a fan-in.** Unless you override it with `graph.SetReducer`, the reducer for a field is inferred from its name prefix, so a field innocently named `listFiles`, `sumTotal`, or `setConfig` is silently appended/added/unioned (not replaced) the moment it is written on more than one parallel branch, producing duplicated or accumulated values that are hard to trace back to the name. For an ordinary field that is not a fan-in accumulator, pick a name that does not start with these prefixes (e.g. `files`, `total`, `config`, `allUsers`). Reach for a prefix only when the task writes a per-branch delta into a field that converges at a `SetFanIn` node.

If a state field's name is dictated by an external schema and cannot follow the prefix convention, attach a reducer explicitly with `graph.SetReducer(field, reducer)`. The supported reducer values are:

- `workflow.ReducerReplace` - last write wins (default)
- `workflow.ReducerAppend` - concatenate arrays
- `workflow.ReducerAdd` - sum numeric values
- `workflow.ReducerUnion` - merge arrays, deduplicate by JSON value
- `workflow.ReducerMerge` - merge objects, new key wins on collision

Explicit reducers are strict: `ReducerUnion` rejects object values, `ReducerMerge` rejects array values. Only the prefix-driven `set*` is polymorphic.

Tasks writing to reducer-managed fields must set only the **delta**, not the full accumulated value. For example, with `listMessages` (auto-append), each fan-out branch should set just the new messages it produced - not the full conversation history.

## Fan-Out and Fan-In

A **fan-out** is any node with two or more normal outgoing transitions, or any `forEach` transition. Goto-only and
error transitions do not count toward fan-out. When a node fans out, the branches it spawns run in parallel, and they
must all converge on a single node that you explicitly mark with `graph.SetFanIn(name)`.

This is mandatory, not optional. `graph.Validate` (called by the workflow's marshaler before the foreman runs the
graph) rejects:

- A fan-out source with no downstream `SetFanIn` node ("fan-out source ... has no fan-in node downstream").
- A path that reaches `workflow.END` while parallel branches are still open ("transition to END ... has unpopped
  fan-out frames").

A graph that fans out but never calls `SetFanIn` compiles and passes `go vet` - it fails only at run time when the
foreman fetches and validates the graph. Adding the fan-out without the matching fan-in is the single most common way
to produce an invalid graph, so add the `SetFanIn` call in the same edit as the fan-out transitions.

```go
graph.AddTask("split", myserviceapi.Split.URL())
graph.AddTask("branchA", myserviceapi.BranchA.URL())
graph.AddTask("branchB", myserviceapi.BranchB.URL())
graph.AddTask("join", myserviceapi.Join.URL())
graph.SetFanIn("join")                  // mark the convergence node
graph.AddTransition("split", "branchA") // fan-out: two normal edges
graph.AddTransition("split", "branchB")
graph.AddTransition("branchA", "join")  // every branch converges on the fan-in node
graph.AddTransition("branchB", "join")
graph.AddTransition("join", workflow.END)
```

The same rule applies to a `forEach` fan-out: `graph.AddTransitionForEach("split", "worker", "items", "item")` followed
by `graph.AddTransition("worker", "join")` and `graph.SetFanIn("join")`. An empty `items` array spawns zero branches;
the foreman then creates the fan-in step directly, so the flow still continues past the join.

State fields produced by the parallel branches are merged at the fan-in node by reducers (see Reducers above), so name
them with the `sum*`, `list*`, or `set*` prefix, or attach a reducer with `SetReducer`. Each branch writes only its own
delta, never the accumulated value.

To reuse one task at two graph positions across a fan-in boundary (for example, fan in and then loop back), register
the same task URL under two distinct names with `AddTask`; each name is a separate node with its own lineage scope. See
`examples/creditflow/service.go` for a worked graph that combines fan-out/fan-in, `forEach`, subgraphs, goto loops, and
error transitions.

## Error Transitions

Error transitions route task failures to an error handler instead of failing the flow. When the source task returns an error, the foreman serializes it as a `*errors.TracedError` (with stack trace, status code, and properties) into the `onErr` state field and dispatches the error handler task. Can be combined with `When` for conditional error routing but not with `ForEach` or `WithGoto`. If the task was in a fan-out, all siblings are cancelled.

The error handler task must include `onErr *errors.TracedError` as an input parameter:

```go
func (svc *Service) HandleMyError(ctx context.Context, flow *workflow.Flow, onErr *errors.TracedError) (result bool, err error) {
    svc.LogWarn(ctx, "Handling error", "error", onErr)
    // Remediation logic...
    return false, nil
}
```

If the error handler itself returns an error, the flow fails normally.

## Subgraphs

A subgraph is a child workflow that executes as a single step in the parent workflow. Register it with `graph.AddSubgraph(workflowName)` and use it as a transition target like any task. On entry, the parent's state is filtered through the child graph's `DeclareInputs`. On completion, the child's final state is filtered through its `DeclareOutputs` before merging into the parent step's changes. This prevents field-name collisions and keeps internal scratch fields from leaking across boundaries.

Interrupts propagate up through the subgraph chain to the root flow. Cancellation cascades down from parent to all child subgraph flows. The parent step is parked until the child completes.

## Dynamic Subgraphs

Static subgraphs are registered at graph-definition time with `graph.AddSubgraph(workflowName)`. Dynamic subgraphs are triggered at runtime by a task calling `flow.Subgraph(workflowURL, input)`. Both follow the same execution model - the only difference is when the parent-child association is established.

`flow.Subgraph` follows the same mental model as `flow.Interrupt`:

| | `flow.Interrupt()` | `flow.Subgraph()` |
|---|---|---|
| **Signal** | Pause, wait for human input | Pause, wait for child workflow |
| **Foreman** | Parks step, waits for `Resume` | Parks step, creates and starts child flow |
| **Interrupts** | This IS the interrupt | Child interrupts propagate up |
| **On completion** | `Resume` sets step to PENDING | Child output merged, step set to PENDING |
| **Re-entry** | Task re-runs with resumed data | Task re-runs with child output in state |

**Re-entry detection**: Before signaling the subgraph, the task should set a flag (e.g. via a typed `Out` suffix argument like `toolExecutedOut = true`). On re-entry, the task checks the flag to know the child completed and processes the result.

**Control signal exclusivity**: A task must set at most one control signal (Goto, Retry, Interrupt, or Subgraph). Setting multiple signals fails the step.

```go
func (svc *Service) MyTask(ctx context.Context, flow *workflow.Flow, executed bool) (executedOut bool, err error) {
    if executed {
        // Re-entry: child workflow completed, its output is in state
        // Process the result...
        return true, nil
    }
    // First run: launch child workflow
    flow.Subgraph(otherserviceapi.SomeWorkflow.URL(), map[string]any{
        "inputField": "value",
    })
    return true, nil
}
```

## Time Budgets and Long-Running Work

Each task step has an execution time budget enforced by the foreman as an HTTP call timeout. The foreman's `TimeBudget` config is the hard ceiling applied to every task dispatch: it defaults to `2m`, and its validation rejects anything outside `1s`-`15m`. A task endpoint declares its own, shorter budget on its subscription with the framework option `sub.TimeBudget(duration)`; the connector then shortens that task handler's context deadline to the smaller of the caller-provided budget and the declared one, so the task is cancelled at its declared bound. The budget is a property of the task endpoint, not of the workflow graph - the graph carries no timing.

If a task exceeds its budget, the foreman cancels the HTTP call and the step's lease eventually expires for `pollPendingSteps` to recover. A task that intrinsically needs more than 15 minutes to finish must not block the worker - instead, use one of the two patterns below to free the worker between work and completion.

### Pattern 1: Interrupt and External Resume

Use when some external signal will indicate completion (a callback from an async service, a webhook arrival, human approval). The task triggers the long-running operation, then calls `flow.Interrupt(payload)` and returns. The worker is freed immediately and the flow parks in `interrupted` status. Some other process (an inbound event sink, a webhook handler, a UI action) calls `foremanapi.Resume(flowID, data)` when the operation completes; the task re-runs with the resume data merged into state.

```go
func (svc *Service) WaitForApproval(ctx context.Context, flow *workflow.Flow, requestID string, approved bool) (approvedOut bool, err error) {
    if approved {
        // Re-entry: Resume was called with approval data merged into state.
        return true, nil
    }
    // First run: kick off the approval request, then park.
    if err := svc.sendApprovalRequest(ctx, requestID); err != nil {
        return false, errors.Trace(err)
    }
    flow.Interrupt(map[string]any{"requestID": requestID})
    return false, nil
}
```

This pattern assumes another process exists that knows when to call `Resume`. Wait time is unbounded - could be seconds or weeks.

### Pattern 2: Polling with Sleep and Retry

Use when no external trigger exists and the task must check for completion itself (waiting on a job ID, a batch import, a slow third-party API). The polling task checks completion; if not done, it calls `flow.Retry` (with backoff) and returns; if done, it returns the result. Each polling cycle is a fresh task execution, so the worker is freed between checks rather than blocking inside `time.Sleep`.

```go
func (svc *Service) PollJobStatus(ctx context.Context, flow *workflow.Flow, jobID string) (jobResult string, err error) {
    status, result, err := svc.checkJobStatus(ctx, jobID)
    if err != nil {
        return "", errors.Trace(err)
    }
    switch status {
    case "running":
        // Not done yet - schedule another poll. Cap attempts to bound total wait.
        if !flow.Retry(120, 30*time.Second, 1, 30*time.Second) { // 120 × 30s = 1h ceiling
            return "", errors.New("job did not complete within poll budget")
        }
        return "", nil // Retry returned true; foreman will re-run this task after the delay.
    case "failed":
        return "", errors.New("job failed")
    default:
        return result, nil
    }
}
```

A separate "kick off the job" task should run before the polling task in the graph, writing `jobID` into state for the poller to read. The two patterns can be combined: trigger work in one task, then either interrupt-and-resume or poll in a downstream task to await completion.

## Running Workflows

The Foreman core service orchestrates workflow execution. Use `foremanapi.NewClient(svc)` to interact with it:

```go
import "github.com/microbus-io/fabric/coreservices/foreman/foremanapi"

// Run a workflow synchronously (create + start + await).
// The final argument is *workflow.FlowOptions; pass nil for defaults.
status, state, err := foremanapi.NewClient(svc).Run(ctx, myworkflowapi.MyWorkflow.URL(), initialState, nil)

// Fine-grained control
flowID, err := foremanapi.NewClient(svc).Create(ctx, myworkflowapi.MyWorkflow.URL(), initialState, nil)
err = foremanapi.NewClient(svc).Start(ctx, flowID)
status, state, err := foremanapi.NewClient(svc).Await(ctx, flowID)

// Inspect a running flow
status, state, err := foremanapi.NewClient(svc).Snapshot(ctx, flowID)

// Cancel a flow
err = foremanapi.NewClient(svc).Cancel(ctx, flowID)

// Resume an interrupted flow with new data
err = foremanapi.NewClient(svc).Resume(ctx, flowID, resumeData)

// Continue a thread with new input (multi-turn pattern)
// The threadKey can be any flowKey belonging to the thread; Continue finds the latest completed flow
newFlowID, err := foremanapi.NewClient(svc).Continue(ctx, threadKey, additionalState)
err = foremanapi.NewClient(svc).Start(ctx, newFlowID)
```

For typed workflow invocation in tests, use the generated `Executor` with a `WorkflowRunner`:

```go
foremanClient := foremanapi.NewClient(tester)
exec := myworkflowapi.NewExecutor(tester).WithWorkflowRunner(foremanClient)
output1, status, err := exec.MyWorkflow(ctx, input1, input2)
```

The `WorkflowRunner` interface is defined in the API package to avoid a `foremanapi` dependency. `foremanapi.Client` satisfies it implicitly.

### Flow Options (priority and fairness)

A flow can carry scheduling hints via `*workflow.FlowOptions` (a flow-level concept, so it lives in the `workflow` package, not `foremanapi`):

```go
opts := &workflow.FlowOptions{
    Priority:       1,        // explicit priority is >= 1, lower runs first; 0 = unset -> the foreman DefaultPriority config
    FairnessKey:    "tenant-42", // "" -> derived from the tid/tenant actor claim, else the "" bucket
    FairnessWeight: 2.0,      // relative dispatch share of the fairness key; 0 -> 1
}
status, state, err := foremanapi.NewClient(svc).Run(ctx, myworkflowapi.MyWorkflow.URL(), initialState, opts)
flowID, err := foremanapi.NewClient(svc).Create(ctx, myworkflowapi.MyWorkflow.URL(), initialState, opts)
```

These three values are immutable for the life of the flow and are inherited unchanged by `Continue`, `Fork`, and subgraphs. The foreman owns all defaulting; pass `nil` (or leave a field zero) to take the default. The `DefaultPriority` config on `foreman.core` sets the priority assigned when the caller specifies none.

The generated per-workflow `Executor` methods keep their minimal signatures. To run a *typed* workflow with options in a test, attach them once with the `WithFlowOptions` builder:

```go
exec := myworkflowapi.NewExecutor(tester).
    WithWorkflowRunner(foremanClient).
    WithFlowOptions(&workflow.FlowOptions{Priority: 1})
output1, status, err := exec.MyWorkflow(ctx, input1, input2) // unchanged signature
```

Scheduling semantics (active): the foreman dispatches by a two-level selection over the pending steps.

- Priority is a strict band. The lowest priority number with due pending work runs first, with no aging and no
  safety valve: a continuously fed stream of higher-priority flows will starve a lower-priority flow for as
  long as the stream lasts. This starvation is by design - use priority only for genuine urgency tiers, not as
  a soft hint.
- Within the chosen priority band, a fairness key is picked by weighted random (share proportional to
  `FairnessWeight`), then that key's oldest pending step runs. The pick is re-rolled per dispatch, so expected
  throughput per key tracks its weight regardless of how deep each key has backlogged - a heavy key cannot
  monopolize a light one. Order within a single key is strict FIFO by creation.
- The weight used for a key is taken from that key's oldest pending step. Supplying inconsistent
  `FairnessWeight` values for the same key across flows is the caller's responsibility (the behavior is
  well-defined but the effective weight is then whichever flow is currently oldest).

Relevant `foreman.core` configs: `DefaultPriority` (priority when the caller specifies none), `Workers` (the
per-replica candidate-cache size and dispatch parallelism), and `TimeBudget` (the hard per-step dispatch
ceiling; a task may declare a shorter `sub.TimeBudget` of its own).

## Flow Stop Notifications

Use `StartNotify` to receive an event when a flow stops (completed, failed, cancelled, or interrupted):

```go
// Start with notification
err = foremanapi.NewClient(svc).StartNotify(ctx, flowID, svc.Hostname())

// Subscribe to notifications
foremanapi.NewHook(svc).ForHost(svc.Hostname()).OnFlowStopped(
	func(ctx context.Context, flowID string, status string, snapshot map[string]any) error {
		// Handle flow completion...
		return nil
	},
)
```

## Debugging Workflows

- `foremanapi.NewClient(svc).BreakBefore(ctx, flowID, taskName, true)` - pause before a task executes
- `foremanapi.NewClient(svc).History(ctx, flowID)` - step-by-step execution trace with state snapshots and nested subgraph history
- `foremanapi.NewClient(svc).Fork(ctx, stepKey, stateOverrides)` - create a new flow from any step checkpoint with modified state

## Mocking Workflows

Workflows can be mocked at two levels of fidelity. Both require the **real foreman** in the test app.

**Tier 1: Workflow-level mock (common case)** - Collapses the entire workflow into a single synthetic task. The handler receives the `*workflow.Flow` carrier with access to state and control signals:

```go
creditflowMock := creditflow.NewMock()
creditflowMock.MockCreditApproval(func(ctx context.Context, flow *workflow.Flow, applicant creditflowapi.Applicant, faultInjection string) (approved bool, creditVerified bool, employmentFailures int, identityVerified bool, err error) {
	return true, true, 0, true, nil
})

app := application.New()
app.Add(svc, foreman.NewService(), creditflowMock, tester)
app.RunInTest(t)
```

Under the hood, `MockCreditApproval` overrides the graph endpoint to return a trivial single-task graph and subscribes a synthetic task handler on the mock. Pass `nil` to clear the mock.

**Tier 2: Individual task mocks (rare case)** - For fine-grained control over specific tasks (interrupt/resume, per-step assertions), mock individual task endpoints while letting the real graph drive execution:

```go
creditflowMock := creditflow.NewMock()
creditflowMock.MockVerifyCredit(func(ctx context.Context, flow *workflow.Flow, creditScore int, faultInjection string) (creditVerified bool, err error) {
	return true, nil
})
```
