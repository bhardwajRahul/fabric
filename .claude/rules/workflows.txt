# Agentic Workflows

Agentic workflows allow microservices to collaborate on multi-step processes. A workflow is a directed graph of tasks orchestrated by the Foreman core service. Each task is a standalone endpoint that reads from and writes to a shared state carried by a `*workflow.Flow`.

## Vocabulary

In this codebase, **"workflow", "agent", and "agentic workflow" all refer to the same thing**: a `workflow.Graph` of tasks orchestrated by the Foreman. The everything in this file applies whether the workflow is rule-based and deterministic, driven by a small specialized model, or built around `llmapi.ChatLoop` and a frontier LLM. An LLM call is *not* a prerequisite for a workflow to be an agent - an agent is autonomous software that perceives state, decides, and acts toward a goal, and a Microbus workflow satisfies that definition regardless of which task types it contains.

So when a user says "create an agent", "create an agentic workflow", or "create a workflow", treat them as the same request and use the `add-workflow` skill. The same applies to "agent step" / "workflow step" / "agentic task" - all map to a single task endpoint and the `add-task` skill. The presence or absence of an LLM-driven task does not change which skill to use, which conventions apply, or which file structure to produce.

Naming convention for agent microservices: a microservice whose sole reason for existing is to expose one workflow is conventionally named `<role>.agent` (e.g. `planner.agent`, `researcher.agent`, `coordinator.agent`). This is just a naming convention, not a framework requirement - the microservice itself is built and registered exactly like any other Microbus microservice.

## Task Endpoints

A task endpoint is a functional endpoint that performs a single step of a workflow. Its signature always includes `ctx context.Context` and `flow *workflow.Flow` as the first two parameters, followed by input arguments sourced from the flow's state, and returns output arguments written back to the state plus `err error`.

```go
import "github.com/microbus-io/dwarf/workflow"

func (svc *Service) VerifyCredit(ctx context.Context, flow *workflow.Flow, creditScore int) (creditVerified bool, err error) {
	if creditScore >= 700 {
		return true, nil
	}
	return false, nil
}
```

Input arguments are matched by name to fields in the flow's state. Output arguments are written back to the state under their name. An output argument with an `Out` suffix (e.g. `creditScoreOut`) implements a read-modify-write pattern - it appears as both input and output.

A task need not have any outputs. A pure side-effect step (restart a node, attach a file, set a status, then `flow.Goto`) returns only `err error`, and its generated `Out` struct is an empty `struct{}`. That is valid and idiomatic - such a task contributes no state fields and routes purely by side effect, `flow.Goto`, or a downstream `Switch`/`When` reading state set by other tasks.

**Prefer typed arguments over `flow.Get`/`flow.Set`.** Task inputs and outputs should be expressed as typed function arguments wherever the types are available in the API package. This makes the task's state contract explicit in the signature and lets the marshaler handle serialization. Reserve `flow.Get`/`flow.Set` only for internal types that are not part of the public API (e.g. intermediate state shared between tasks within the same microservice).

```go
// Typed arguments make the state contract explicit
func (svc *Service) ProcessResponse(ctx context.Context, flow *workflow.Flow, llmContent string, toolRounds int) (toolsRequested bool, toolRoundsOut int, err error) {
    // llmContent and toolRounds are read from state automatically
    // toolsRequested and toolRoundsOut are written back to state automatically
}
```

Tasks are registered on port `:428` by default.

## Workflow Graph Endpoints

A workflow graph endpoint returns a `*workflow.Graph` that defines the structure of a workflow - its tasks, transitions, and merge strategies. The graph is fetched once when a flow is created and stored with it.

**Naming convention: graph names and task (node) names are PascalCase** (`CreditApproval`, `VerifyCredit`) - they are graph-topology identifiers, kept visually distinct from the lowercased dispatch URLs the nodes resolve to (`myserviceapi.VerifyCredit.URL()`) and the camelCase state field names (`creditVerified`). `SetEndpoint(name, url)` registers a node under a PascalCase name that resolves to the task's URL; every transition, `SetFanIn`, and `flow.Goto` then refers to that PascalCase node name, never the URL.

```go
import "github.com/microbus-io/dwarf/workflow"

func (svc *Service) CreditApproval(ctx context.Context) (graph *workflow.Graph, err error) {
	graph = workflow.NewGraph("CreditApproval")
	// SetEndpoint(PascalCaseNodeName, dispatchURL): the node name is the graph identifier; the URL is where it runs.
	graph.SetEndpoint("SubmitApplication", myserviceapi.SubmitApplication.URL())
	graph.SetEndpoint("VerifyCredit", myserviceapi.VerifyCredit.URL())
	graph.SetEndpoint("VerifyEmployment", myserviceapi.VerifyEmployment.URL())
	graph.SetEndpoint("FinalDecision", myserviceapi.FinalDecision.URL())

	// Fan-out: SubmitApplication has two normal outgoing transitions, so VerifyCredit and VerifyEmployment run as parallel branches
	graph.AddTransition("SubmitApplication", "VerifyCredit")
	graph.AddTransition("SubmitApplication", "VerifyEmployment")
	// Fan-in: every parallel branch must converge on a node marked with SetFanIn, or graph.Validate rejects the graph
	graph.SetFanIn("FinalDecision")
	graph.AddTransition("VerifyCredit", "FinalDecision")
	graph.AddTransition("VerifyEmployment", "FinalDecision")
	// Conditional transition: the when expression references a state field, which stays camelCase
	graph.AddTransitionWhen("FinalDecision", workflow.END, "creditVerified")
	// Goto transition (taken only when the task calls flow.Goto), targeting a PascalCase node name
	graph.AddTransitionGoto("FinalDecision", "SubmitApplication")

	return graph, nil
}
```

Task URLs in the graph are obtained from the generated API package constants (e.g. `myserviceapi.TaskName.URL()`); the node names you pass alongside them are PascalCase labels you choose (conventionally matching the task's name).

## The `workflow` Package

**`workflow.Flow`** is the carrier passed to every task. It provides typed accessors for reading and writing state:

- Read: `GetString`, `GetInt`, `GetFloat`, `GetBool`, `GetDuration`, `GetStrings`, `Has`, `Get` (for complex types), `ParseState` (unmarshal into struct)
- Write: `SetString`, `SetInt`, `SetFloat`, `SetBool`, `SetDuration`, `SetStrings`, `Set` (for complex types)
- Struct-based: `Snapshot` captures state before modifications, `SetChanges(source, snap)` records only fields that differ from the snapshot

Control signals on `Flow`:

- `flow.Goto(taskName)` - override transition routing to jump to a specific task node (its PascalCase node name, not its URL)
- `flow.Retry(initialDelay, delayMultiplier, maxIntervalDelay, giveUpAfter) bool` - re-execute this task with exponential backoff. The bound is **wall-clock, not a count**: returns true while the next attempt would still land within `giveUpAfter` of the step's creation, false once the horizon is reached (including when the next backoff delay alone would overshoot it, so a doomed wait is never parked before failing). The delay before attempt N is `min(initialDelay * delayMultiplier^N, maxIntervalDelay)`; immediate retry is a zero `initialDelay`, fixed interval is `delayMultiplier` 1.0, no per-interval cap is a zero `maxIntervalDelay`. Pass `giveUpAfter <= 0` for unlimited retry; to bound by count instead, pass `0` and gate on `flow.Attempt()`. The single retry primitive: it carries no condition of its own, so the task writes the retryable predicate explicitly in the surrounding `if`. Timeout-only retry is `if errors.StatusCode(err) == http.StatusRequestTimeout && flow.Retry(...)` - both foreman-side `pub.Timeout` and subscriber-side handler timeouts surface as 408
- `flow.Sleep(duration)` - delay the next step by a duration
- `flow.Interrupt(payload, &resume) (yield, err)` - park the step for external input (human-in-the-loop). First call returns `yield=true` and the task must `return nil` immediately; on re-entry after `Resume(flowKey, data)` returns `(false, nil)` with `data` unmarshaled into `&resume` so the task proceeds (see [Subgraphs and Interrupts](#subgraphs-and-interrupts))
- `flow.Subgraph(workflowURL, in, &out) (yield, err)` - park the step and launch a child *workflow*. First call returns `yield=true` (task must `return nil`); on re-entry returns `(false, nil)` with the child's `final_state` unmarshaled into `&out` on success, or `(false, childErr)` on failure - the task adopts fields from `out` or branches on `err`. Prefer the typed `xapi.NewSubgraph(flow)` client over the raw primitive (see [Subgraphs and Interrupts](#subgraphs-and-interrupts))

**`workflow.Graph`** defines workflow structure:

- `NewGraph(name)` - create a new graph (name is a human-friendly PascalCase display label, surfaced in rendering). The graph does NOT carry a resolve URL: the resolve key is a separate opaque URL passed to the foreman (`Run`/`Create`/`LoadGraph`), never stored on the graph
- `SetEndpoint(name, taskURL)` - bind a node (by PascalCase name) to its dispatch URL, create-or-update (first node bound becomes the default entry point; the same URL may be bound under multiple names)
- `SetEntryPoint(name)` - override the default entry point by node name
- `AddTransition(from, to)` - unconditional transition; sibling unconditional/`When` transitions from the same node all fire in parallel (fan-out)
- `AddTransitionChain(names...)` - convenience for a linear path; adds an unconditional `AddTransition` between each consecutive pair (`AddTransitionChain("A", "B", "C")` == `AddTransition("A", "B")` + `AddTransition("B", "C")`). End a chain with `workflow.END`. For branching, conditions, and loops, mix with the other `AddTransition*` methods
- `AddTransitionWhen(from, to, when)` - conditional transition based on state expression; sibling `When` transitions are evaluated independently and *all* matching ones fire (fan-out, not first-match)
- `AddTransitionSwitch(from, to, when)` - first-match-wins routing; siblings are evaluated in registration order and only the first whose expression matches fires. Use `when="true"` as the last branch for a default. Only one branch ever runs, so no `SetFanIn` is needed. See [Switch Transitions](#switch-transitions) below
- `AddTransitionGoto(from, to)` - transition taken only when `flow.Goto` is called from the task body
- `AddTransitionForEach(from, to, forEach, as)` - dynamic fan-out over a state array field (`from`/`to` are PascalCase node names; `forEach`/`as` are camelCase state fields)
- `SetFanIn(name)` - mark a node (by PascalCase name) as the fan-in nexus where parallel branches converge. Mandatory whenever the graph fans out: every fan-out source (a node with 2+ normal outgoing transitions, or any `forEach` transition) must converge on a `SetFanIn` node, otherwise `Validate` rejects the graph (see [Fan-Out and Fan-In](#fan-out-and-fan-in)). `Switch` transitions do **not** count as fan-out and need no `SetFanIn`
- `AddTransitionOnError(from, to)` - transition taken when the source task returns an error (see [Error Transitions](#error-transitions))
- `SetReducer(field, reducer)` - attach a fan-in merge strategy to a state field (mandatory whenever a fan-in field needs anything other than last-write-wins; see Reducers below)
- `workflow.END` - pseudo-node indicating workflow termination

**Reducers** control how state fields are merged at fan-in points. Every field that needs anything other than last-write-wins must be wired explicitly with `graph.SetReducer(field, reducer)` at graph-build time. Fields without a registered reducer use `Replace` (last write wins). State field names carry no hidden semantics - name them naturally (e.g. `messages`, `total`, `tags`) and attach the reducer separately.

```go
graph.SetReducer("messages", workflow.ReducerAppend)  // accumulate per-branch deltas
graph.SetReducer("total",    workflow.ReducerAdd)     // sum numeric contributions
graph.SetReducer("seen",     workflow.ReducerUnion)   // dedupe across branches
graph.SetReducer("approved", workflow.ReducerAnd)     // all branches must approve
graph.SetReducer("flagged",  workflow.ReducerOr)      // any branch flags
graph.SetReducer("notes",    workflow.ReducerConcat)  // join string deltas
graph.SetReducer("attrs",    workflow.ReducerMerge)   // merge per-branch objects
```

The supported reducer values:

- `workflow.ReducerReplace` - last write wins (default; identity none)
- `workflow.ReducerAppend` - concatenate arrays (identity `[]`)
- `workflow.ReducerAdd` - sum numeric values (identity `0`)
- `workflow.ReducerUnion` - merge arrays, deduplicate by JSON value (identity `[]`)
- `workflow.ReducerMerge` - merge objects, new key wins on collision (identity `{}`)
- `workflow.ReducerAnd` - logical AND of booleans (identity `false`)
- `workflow.ReducerOr` - logical OR of booleans (identity `false`)
- `workflow.ReducerConcat` - concatenate strings (identity `""`)

Reducers are strict on their input type. `ReducerUnion` rejects object values, `ReducerMerge` rejects array values, `ReducerAdd` rejects strings, etc. A cleared slot (Go nil or JSON null) short-circuits to the reducer's identity (the Go zero value for the underlying type) rather than failing the type check, so `flow.Clear` contributions are ignored at fan-in.

**Empty-cohort behavior.** A `forEach` that spawns zero branches still fires the fan-in. The field takes the reducer's identity. Both `ReducerAnd` and `ReducerOr` return `false` on an empty cohort - there is no "vacuously true" mode. If you need an empty cohort to be treated as approval, branch on the cohort size downstream rather than relying on the reducer.

**Order sensitivity.** `Append` (arrays) and `Concat` (strings) preserve the cohort merge order, which is `updated_at, step_id` - not creation order under shard scatter. If you need a specific ordering, sort downstream rather than relying on fan-in order.

Tasks writing to reducer-managed fields must set only the **delta**, not the full accumulated value. For example, with `messages` attached to `ReducerAppend`, each fan-out branch should set just the new messages it produced - not the full conversation history.

## Fan-Out and Fan-In

A **fan-out** is any node with two or more normal outgoing transitions, or any `forEach` transition. Goto-only and
error transitions do not count toward fan-out. When a node fans out, the branches it spawns run in parallel, and they
must all converge on a single node that you explicitly mark with `graph.SetFanIn(name)`.

This is mandatory, not optional. `graph.Validate` (called by the workflow's marshaler before the foreman runs the
graph) rejects:

- A fan-out source with no downstream `SetFanIn` node ("fan-out source ... has no fan-in node downstream").
- A path that reaches `workflow.END` while parallel branches are still open ("transition to END ... has unpopped
  fan-out frames").

A graph that fans out but never calls `SetFanIn` compiles and passes `go vet` - it fails only at run time when the
foreman fetches and validates the graph. Adding the fan-out without the matching fan-in is the single most common way
to produce an invalid graph, so add the `SetFanIn` call in the same edit as the fan-out transitions.

```go
graph.SetEndpoint("Split", myserviceapi.Split.URL())
graph.SetEndpoint("BranchA", myserviceapi.BranchA.URL())
graph.SetEndpoint("BranchB", myserviceapi.BranchB.URL())
graph.SetEndpoint("Join", myserviceapi.Join.URL())
graph.SetFanIn("Join")                  // mark the convergence node
graph.AddTransition("Split", "BranchA") // fan-out: two normal edges
graph.AddTransition("Split", "BranchB")
graph.AddTransition("BranchA", "Join")  // every branch converges on the fan-in node
graph.AddTransition("BranchB", "Join")
graph.AddTransition("Join", workflow.END)
```

The same rule applies to a `forEach` fan-out: `graph.AddTransitionForEach("Split", "Worker", "items", "item")` followed
by `graph.AddTransition("Worker", "Join")` and `graph.SetFanIn("Join")` (node names PascalCase; `items`/`item` are state fields). An empty `items` array spawns zero branches;
the foreman then creates the fan-in step directly, so the flow still continues past the join.

State fields produced by the parallel branches are merged at the fan-in node by reducers (see Reducers above). Attach
the appropriate reducer at graph-build time with `graph.SetReducer(field, reducer)`. Each branch writes only its own
delta, never the accumulated value.

### `forEach` Branch State

A `forEach` branch does **not** receive a copy of the source array. The foreman strips the array (`forEach` field
name) from each branch's local state at spawn, so an N-element forEach over a sub-chain `A -> B -> C` does not write
N copies of the array across every step in every branch. The array still lives in the spawn step's immutable
snapshot, and the fan-in step rebuilds its own state from that snapshot - so the array is absent only **inside the
cohort** and reappears in the fan-in step and everything downstream of it. A branch task that needs to read
something from the source array must take it as an explicit per-element value (the `as` argument) or look it up
some other way.

Alongside the strip, the foreman injects two read-only fields on each branch's state, named after the `as` alias:

- `<as>Index` (`int`) - this element's 0-based position in the `forEach` array.
- `<as>Count` (`int`) - the cohort size (total elements in the `forEach` array).

For `AddTransitionForEach("Split", "Worker", "items", "item")` the branch sees `item`, `itemIndex`, `itemCount`.
A task argument by that name (e.g. `func Worker(... item string, itemIndex int, itemCount int)`) is auto-populated
from state like any other. These fields are present only for `forEach` fan-out, not static fan-out.

`<as>`, `<as>Index`, and `<as>Count` are also **dropped from the merged state at fan-in**, symmetric to the source-
array strip on the way down. They are per-element bookkeeping with no meaning past the cohort, so the foreman
deletes them after merging the cohort's contributions. A workflow that needs the element value past the fan-in
must forward it under a different key in the branch (e.g. `flow.SetString("chosenItem", item)`); the `as` alias
itself is gone downstream of the fan-in.

To suppress the source array past the fan-in (useful when it is very large and only the per-element transformation
matters downstream), have the branch call `flow.Set("<forEach>", nil)` in its task body. That writes the new value
into the branch's `changes`, the replace reducer folds it over the spawn-step base at fan-in, and the field is
absent (or whatever the branch wrote) past the join.

To reuse one task at two graph positions across a fan-in boundary (for example, fan in and then loop back), register
the same task URL under two distinct names with `AddTask`; each name is a separate node with its own lineage scope. See
`examples/creditflow/service.go` for a worked graph that combines fan-out/fan-in, `forEach`, subgraphs, goto loops, and
error transitions.

## Switch Transitions

`AddTransitionSwitch(from, to, when)` declares a first-match-wins outgoing transition. When the source task finishes, the foreman walks the source's `Switch` transitions in declaration order, evaluates each `when` expression against the post-task state, and follows the **first** one that matches. The remaining `Switch` transitions are skipped on that step. Because exactly one branch ever fires, the downstream branches do **not** need to converge on a `SetFanIn` node.

```go
graph.SetEndpoint("Router", myserviceapi.Router.URL())
graph.SetEndpoint("HighValue", myserviceapi.HighValue.URL())
graph.SetEndpoint("MidValue", myserviceapi.MidValue.URL())
graph.SetEndpoint("LowValue", myserviceapi.LowValue.URL())
graph.AddTransitionSwitch("Router", "HighValue", "amount >= 10000")
graph.AddTransitionSwitch("Router", "MidValue", "amount >= 1000")
graph.AddTransitionSwitch("Router", "LowValue", "true") // default branch
graph.AddTransition("HighValue", workflow.END)
graph.AddTransition("MidValue", workflow.END)
graph.AddTransition("LowValue", workflow.END)
```

Rules the validator enforces:

- `when` is required on every `Switch` transition. Use `"true"` for the default catch-all.
- A node that uses `Switch` must declare **every** success-path outgoing transition as `Switch`. Mixing `Switch` with `When`, plain `AddTransition`, `AddTransitionForEach`, or `AddTransitionGoto` from the same source is rejected. `AddTransitionOnError` stays allowed because they live on a different axis (error vs. success).
- `Switch` cannot combine with `ForEach`, `WithGoto`, or `OnError` on the same transition.

If no `Switch` matches at runtime, the flow ends at the source node (no successor is created, the source step completes, and the flow transitions to `completed` via the normal "no next steps" path). To make termination explicit, target `workflow.END` from a final `Switch` with a tail predicate.

### When to choose Switch vs. When vs. Goto

Prefer in this order:

1. **`AddTransitionSwitch`** for any routing where you want exactly one branch to run based on data the graph can see. It's the safest of the three: the validator guarantees mutual exclusivity (so two siblings can't accidentally both fire and leak a parallel branch), there's no `SetFanIn` to forget downstream, and the Mermaid diagram labels each arm with its predicate. The cost is rigidity: every success-path edge from the source must be `Switch` once any of them is, so you commit the whole node to switch-style routing.
2. **`AddTransitionWhen`** when the source genuinely **does** fan out under multiple matching conditions, or when one of the siblings is unconditional and represents a "do this in addition to" branch. `When` predicates are evaluated independently and any number of them can match - that's the right semantic for parallel work, the wrong one for routing. If you're writing two `AddTransitionWhen` calls whose predicates are intended to be mutually exclusive (`x>0` and `x<=0`), convert them to `Switch` so the validator enforces what you meant; non-exclusive `When` predicates are a footgun (both fire, you suddenly need `SetFanIn`).
3. **`AddTransitionGoto`** only for runtime control-flow decisions the task body makes - loop-back from a fan-in node, "ask for more input then retry", LLM tool loops. `Goto` is invisible to the graph's structural rules (no lineage push/pop), so reach for it only when the branching condition is not a function of state that the graph could evaluate.

In short: `Switch` is the default for branching. Use `When` only for genuine fan-out, `Goto` only when the task body must decide at runtime.

## Error Transitions

Error transitions route task failures to an error handler instead of failing the flow. When the source task returns an error, the foreman serializes it as a `*errors.TracedError` (with stack trace, status code, and properties) into the `onErr` state field and dispatches the error handler task. Can be combined with `When` for conditional error routing but not with `ForEach` or `WithGoto`. If the task was in a fan-out, all siblings are cancelled.

The error handler task must include `onErr *errors.TracedError` as an input parameter:

```go
func (svc *Service) HandleMyError(ctx context.Context, flow *workflow.Flow, onErr *errors.TracedError) (result bool, err error) {
    svc.LogWarn(ctx, "Handling error", "error", onErr)
    // Remediation logic...
    return false, nil
}
```

If the error handler itself returns an error, the flow fails normally.

## Subgraphs and Interrupts

A **subgraph** is a child *workflow* launched from inside a task body via `flow.Subgraph(workflowURL, in, &out)`. The parent step parks while the child runs and re-enters the task once the child terminates. **This is the only correct way for one task to invoke another unit of work with state isolation - do not call `foremanapi.NewClient(svc).Run`, nor the test-only `Executor`, from a task body; that spawns an independent flow (or a foreman-coupled call) and forfeits re-entry, single-flow audit trail, cascading cancellation, and interrupt propagation.**

A bare *task* is not an independently invocable unit - it is only ever a node in a graph. To invoke a single unit of work in isolation, declare a one-node workflow and subgraph it.

**Two ways to compose tasks, by state exposure.** Inside a flow you reach task B either by **graph composition** (B is a node in the same graph: it sees and writes the parent's *full* state - tightest coupling) or by a **subgraph** (B is a separate workflow run as an isolated child: only the explicit `in` crosses in, only the explicit `out` crosses back). Use graph composition when B is a genuine step of this workflow sharing its state vocabulary; use a subgraph when B is a reusable or foreign workflow you invoke with explicit arguments and don't want to hand your whole state to.

**A subgraph is a function call.** Only the explicit `in` crosses into the child as its initial state, and only the explicit `out` (the child's `final_state`, unmarshaled into your pointer by JSON tag) crosses back. The parent's state is NOT inherited in either direction. `nil` input means "no arguments". A caller that wants the parent's full state to cross passes `flow.Snapshot()` (or a derived map) as `in`.

**Prefer the typed client over the raw primitive.** The generated `xapi.NewSubgraph(flow)` client wraps `flow.Subgraph` with typed In/Out: `out1, yield, err := otherapi.NewSubgraph(flow).SomeWorkflow(ctx, args...)`. Reach for the raw `flow.Subgraph` only for dynamic/untyped dispatch (a computed URL, or a service whose `*api` package you don't import).

### The park-and-resume pattern

`flow.Subgraph` and `flow.Interrupt` share one shape: park-the-step now, return the resolved value on re-entry. The result is unmarshaled into a trailing `out` pointer (`*struct` for type safety, `*map[string]any` for dynamic access, or `nil` to ignore); the call returns only `(yield, err)`.

```go
// Subgraph: embed a whole child workflow.
var out SomeOut // or: var out map[string]any
yield, err := flow.Subgraph(otherapi.SomeWorkflow.URL(), SomeIn{Field: "value"}, &out)
if yield {
    return nil          // first pass: parked, child running
}
if err != nil {
    // Child failed. The task may inspect err and retry, route, or propagate.
    if errors.StatusCode(err) == http.StatusRequestTimeout && flow.Retry(time.Second, 2.0, 30*time.Second, 5*time.Minute) {
        return nil      // re-arms a fresh child on re-dispatch
    }
    return errors.Trace(err)
}
// Resolved + succeeded: adopt the fields the parent wants from out into its own state.
flow.SetString("parentResult", out.Field)
return nil
```

`flow.Interrupt` is symmetric (its `out` carries the `Resume` payload):

```go
var resume map[string]any
yield, err := flow.Interrupt(map[string]any{"request": "userInput"}, &resume)
if yield {
    return nil          // first pass: parked, waiting for Resume
}
// resume carries the Resume payload; err is always nil for Interrupt
```

| | `flow.Interrupt()` | `flow.Subgraph()` |
|---|---|---|
| **Signal** | Park, wait for external `Resume` | Park, wait for child workflow |
| **Foreman** | Parks the step, sets `interrupt_done=0` | Parks the step, launches child, sets `subgraph_done=0` |
| **On completion** | `Resume(flowKey, data)` writes `resume_data` and re-dispatches | Child terminal outcome writes `subgraph_result`/`subgraph_error` and re-dispatches |
| **Re-entry signal** | `yield=false`, `resumeData` is the Resume payload | `yield=false`, `out` is the child's `final_state` (success) or `err` is the child's failure |
| **Failure mode** | None (always succeeds on Resume) | Child failure delivered to parent as `err`; parent decides retry/route/propagate |

**Re-entry is signaled by the parker's `yield` bool, not by a state field.** Do not invent a `*Out`-suffix flag or a "have I run before" arg - the framework already tracks it via `interrupt_done`/`subgraph_done` columns and surfaces it as `yield`. The `if yield { return nil }` line is the **before-resume region** and runs on every dispatch (first pass, re-entry, every retry attempt); everything after it is the **after-resume region** and runs once per resolution. Exactly-once side effects (firing an event, charging a card) belong after the guard.

**State written before the park survives the park.** A `flow.Set*` / `flow.Delete` made in the same dispatch *before* calling `flow.Interrupt` (or `flow.Subgraph`) is recorded in the step's `changes` and persisted when the step parks, so it is visible as an input on re-entry and to downstream tasks. This is the basis for idempotent side-effecting parks: a task that creates an external resource, records its identifier with `flow.SetString`, and then parks reads that identifier back on re-entry and skips re-creating the resource (e.g. `if id == "" { id = create(); flow.SetString("id", id) }; flow.Interrupt(...)`). Persisting through the change set is what makes this work - a plain local variable, or relying on a return value (output args are written as their zero value when you `return nil` on `yield`), would *not* survive the park.

**A step parks at most once per dispatch, interrupt XOR subgraph, never both and never two of the same kind.** Sequential side-trips are sequential tasks in the graph, not multiple parks in one task. The guard fires in two places: the `Flow` rejects a second park synchronously with an error from the parker; the foreman re-checks the returned flow after the task returns (trust boundary for a bypassed parker).

**No auto-merge.** The child's `final_state` is returned to the calling task as `out`, not merged into the parent's state. This is the "scope state at the subgraph boundary" payoff: the parent picks fields explicitly with `flow.Set`, `flow.Transform`, or just plain Go code over `out`. The calling task does the reshape itself, so the common case needs no adapter task upstream or downstream of the subgraph.

**Interrupts propagate up through the subgraph chain to the root flow.** A child subgraph can call `flow.Interrupt`; the foreman walks the surgraph chain up to the root, the external awaiter of the root flow sees the interrupt with the child's payload, and `Resume` propagates back down to the leaf. The parent's surgraph step stays parked across the whole interrupt/resume cycle; the parent task does **not** re-run for it. This is orthogonal to `subgraph_done` (interrupt rides the step status; `subgraph_done` flips only on terminal completion or failure).

**Cancellation cascades the whole tree.** Cancelling a child cancels its parents. There is no "recover from a bad child" via cancel-of-child; that need is served by the failure-as-`err` path (child timeout surfaces as `err` with 408, parent does `flow.Retry`, an `OnError` route, or propagates).

**Single control signal per dispatch.** A task must set at most one of `Goto`, `Retry`, `Interrupt`, or `Subgraph`. Setting multiple fails the step.

### State Transformation Around a Subgraph

`in` scopes what crosses *into* the child; `out` scopes what comes back. If the parent and child use different field names, reshape inline: build `in` with the names the child expects, then read `out` with the names the child wrote. The `flow.Transform` / `Delete` / `Clear` primitives help when the reshape is non-trivial (see [Best Practices](#best-practices)). The calling task does the reshape itself; no dedicated adapter task is involved.

## Using LLMs in Workflows

There is exactly one way to use an LLM inside a workflow: **compose `llmapi.ChatLoop` as a subgraph**
with a *prepare* task immediately upstream and a *handle* task immediately downstream. Do not call
`llmapi.Chat` from a task body (it blocks one worker for the entire multi-turn conversation, with no
foreman persistence between turns) and do not call `llmapi.Turn` directly (it is the raw single-turn
primitive each provider microservice implements; you would be bypassing the tool-calling loop, usage
aggregation, and re-entry semantics that `ChatLoop` provides).

### The Two-Task Shape

`ChatLoop` is launched as a dynamic subgraph from a caller task. A second task downstream reads
the conversation result. The two tasks are connected by a single `AddTransition`; the subgraph is
not a graph node.

```go
func (svc *Service) MyWorkflow(ctx context.Context) (graph *workflow.Graph, err error) {
    graph = workflow.NewGraph("MyWorkflow")
    graph.SetEndpoint("RunChat", myapi.RunChat.URL())               // calls flow.Subgraph(llmapi.ChatLoop, ...)
    graph.SetEndpoint("HandleResponse", myapi.HandleResponse.URL()) // reads the final assistant message
    graph.AddTransition("RunChat", "HandleResponse")
    graph.AddTransition("HandleResponse", workflow.END)
    return graph, nil
}
```

Every turn (`CallLLM`, `ProcessResponse`, each `ExecuteTool`) becomes its own persisted foreman step,
so a worker crash mid-conversation resumes at the last completed step, the full transcript appears in
the audit trail, and the subgraph releases the worker between turns. No worker is ever pinned for the
whole loop.

### The *Run* Task

The run task builds the `ChatLoop` input, calls `flow.Subgraph`, and adopts the conversation result
from `out` on re-entry. The five input fields are the `ChatLoop` input contract:

- `provider string` - target provider hostname (e.g. `claudellmapi.Hostname`, `chatgptllmapi.Hostname`,
  `geminillmapi.Hostname`).
- `model string` - model identifier from the provider's `*api` constants (e.g.
  `claudellmapi.ModelHaiku45`, `chatgptllmapi.ModelGPT4o`). Typed constants make a typo a compile
  error.
- `messages []llmapi.Message` - the initial conversation (system prompt + user turn + any prior
  history the workflow is replaying).
- `toolURLs []string` - canonical Microbus endpoint URLs the LLM may call (see "Exposing Tools"
  below). Empty or nil disables tool calling. `InitChat` resolves each URL to a tool schema by
  fetching its host's OpenAPI document at the start of the conversation.
- `options *llmapi.ChatOptions` - optional `MaxTokens`, `Temperature`, `MaxToolRounds`. `nil`
  accepts defaults.

```go
func (svc *Service) RunChat(ctx context.Context, flow *workflow.Flow, userQuestion string) (
    messages []llmapi.Message, usage llmapi.Usage, err error,
) { // MARKER: RunChat
    var resp llmapi.ChatLoopOut // the child's final_state unmarshals into resp by JSON tag
    yield, err := flow.Subgraph(llmapi.ChatLoop.URL(), map[string]any{
        "provider": claudellmapi.Hostname,
        "model":    claudellmapi.ModelHaiku45,
        "messages": []llmapi.Message{
            {Role: "system", Content: "You are a helpful assistant."},
            {Role: "user", Content: userQuestion},
        },
        "toolURLs": []string{
            // canonical endpoint URLs the LLM is allowed to call - see "Exposing Tools" below.
            // e.g. calculatorapi.Arithmetic.URL(),
        },
        "options": &llmapi.ChatOptions{MaxTokens: 1024},
    }, &resp)
    if yield {
        return nil, llmapi.Usage{}, nil
    }
    if err != nil {
        return nil, llmapi.Usage{}, errors.Trace(err)
    }
    return resp.MessagesOut, resp.Usage, nil
}
```

### The *Handle* Task

The handle task receives the conversation result via its typed arguments, populated by the run
task's outputs. The two fields are the `ChatLoop` output contract:

- `messages []llmapi.Message` - the full conversation appended in turn order: the original
  system/user prompt, every assistant reply, every tool result, and the final assistant reply that
  ended the loop. The subgraph's internal `Append` reducer assembles this across the per-tool-call
  fan-in; the parent receives one merged value.
- `usage llmapi.Usage` - aggregated token usage across every turn (input, output, cache-read,
  cache-write tokens, plus the resolved model and turn count).

The last message in `messages` is the LLM's final answer:

```go
func (svc *Service) HandleResponse(ctx context.Context, flow *workflow.Flow,
    messages []llmapi.Message, usage llmapi.Usage,
) (answer string, totalTokens int, err error) { // MARKER: HandleResponse
    if len(messages) == 0 {
        return "", 0, errors.New("empty conversation")
    }
    last := messages[len(messages)-1]
    if last.Role != "assistant" {
        // The loop exited mid-conversation (round limit hit, etc.); decide policy.
        return "", 0, errors.New("conversation did not end with an assistant message")
    }
    return last.Content, usage.InputTokens + usage.OutputTokens, nil
}
```

If the parent workflow only needs the final answer, persist that field and `flow.Delete("messages")`
to drop the verbose conversation history from downstream state and shrink the audit trail.

### Exposing Tools

Each entry in `toolURLs` is the canonical URL of a Microbus endpoint the LLM is allowed to call
mid-conversation, taken from the endpoint's `Def` (e.g. `calculatorapi.Arithmetic.URL()`). Typed
constants from each microservice's `*api` package make a typo a compile error and let the prepare
task concatenate URLs across multiple `*api` packages without collision. `InitChat` resolves each
URL to a tool schema by fetching its host's OpenAPI document, so the prepare task only chooses
*which* endpoints to expose, never how to describe them.

Choose the endpoint type by what the tool needs to do:

- **Functional endpoints** (`FeatureFunction` on `:443`). The default. A typed input/output function
  that performs a single, atomic, bounded-latency operation - a calculator, a DB lookup, a
  classifier, a unit converter. The LLM service dispatches the call as a direct bus request and
  returns the function's output to the next turn.
- **Workflow endpoints** (`FeatureWorkflow` on `:428`). Use when the tool itself needs the foreman
  machinery: bounded retries on transient failures (`flow.Retry`), error routing to a fallback
  task (`AddTransitionOnError`), per-step time budgets, a multi-step sequence, or a nested
  subgraph. The LLM service dispatches the call as a **dynamic subgraph**, so every step inside the
  tool's workflow appears in the parent flow's audit trail and re-entry/replay works the same as
  for any other subgraph. Reach for this when "the tool sometimes fails and the LLM shouldn't see
  the failure" or "the tool itself orchestrates multiple steps."

Only `FeatureFunction`, `FeatureWeb`, and `FeatureWorkflow` endpoints can be exposed as tools.
Tasks and outbound events are filtered out at the OpenAPI document level, so a URL pointing at
either is silently skipped during resolution.

Authorization for tool calls piggybacks on the calling actor: `InitChat` runs with the workflow's
actor context, so the host's OpenAPI handler omits operations the actor cannot satisfy and a tool
whose endpoint declares `requiredClaims` will simply disappear from the LLM's tool list.

When two endpoints across the supplied URLs share the same operation name, the first keeps the
bare name and subsequent ones get `_2`, `_3`, ... suffixes in argument order.

### Notes

- **The `messages` boundary field.** The conversation crosses the subgraph boundary as a field
  named `messages`, with its `Append` reducer wired by an explicit `SetReducer` inside the subgraph.
  The parent itself normally needs no reducer on `messages` - the subgraph is the only contributor,
  so last-write-wins is correct. Only when the parent has multiple branches that *also* write to
  `messages` and converge on a fan-in does the parent need
  `graph.SetReducer("messages", workflow.ReducerAppend)`. If the parent carries the conversation
  under a different field name, reshape it in an upstream task with
  `flow.Transform("messages", "yourField")` (see "State Transformation Around a Subgraph" above).
- **Mocking.** Mock the underlying provider's `Turn` (`claudellm.NewMock().MockTurn(handler)`,
  etc.) rather than mocking `ChatLoop` itself - the loop logic, tool dispatch, and usage
  aggregation stay real, and the test exercises the same code path as production.

## Best Practices

A short list of high-leverage choices that emerge from operating these workflows in practice. None are enforced by the validator; they fall to the author.

**Prefer typed input/output arguments over `flow.Get` / `flow.Set`.** The task signature *is* the state contract - the marshaler binds inputs from state on entry and writes outputs back on exit, mocks get typed handlers, and a reader sees what the task reads and produces without scanning the body. Reach for `flow.Get` / `flow.Set` only when a key's name is dynamic, or when the value's type is internal to the microservice (not part of the API package and not worth promoting). The output `Out` suffix convention covers the read-modify-write case (`counter` in, `counterOut` out, both mapped to state key `counter`).

**Use `flow.Delete` / `flow.Clear` / `flow.Transform` to drop state the rest of the flow does not need.** LLM responses, parsed documents, raw API bodies, image bytes - anything a task consumes once and would otherwise ride through every downstream step's persisted snapshot and bloat both the wire and the audit trail. The three primitives compose for any cleanup pattern:

- `flow.Delete(names...)` - drop the listed fields (variadic; mirrors `delete(m, k)`).
- `flow.Clear()` - drop every field (mirrors `clear(m)`); useful in a task that is about to populate a fresh subgraph input from scratch.
- `flow.Transform("newKey", "oldKey", ...)` - clear all state, then re-introduce the listed fields under new names (pairs of `newKey, oldKey`). Convenient for tasks that reshape state across a contract boundary; doubles as a "keep these" primitive when you pass `("name", "name")` pairs.

All three record JSON null in the step's changes for dropped fields, so the cleanup is preserved in the audit trail. For Replace-reduced fields, null drops the field from merged state at the next fan-in. For reducer-managed fields (those wired with `SetReducer`), a cleared contribution is treated as the reducer's identity and contributes nothing to the merge. None of these methods are restricted to boundaries - call them mid-flow whenever a field has served its purpose.

**Prefer `Switch` > `When` > `Goto` for branching.** `AddTransitionSwitch` (first-match-wins, validator-enforced mutual exclusivity, no `SetFanIn` needed) is the right tool for any routing where exactly one branch should run. Drop to `AddTransitionWhen` only when you genuinely want parallel branches firing under independent predicates - `When` is fan-out, not routing, and two `When` predicates intended to be mutually exclusive should be a `Switch` instead. Reserve `AddTransitionGoto` for runtime loops the task body must decide (fan-in looping back, "ask for more info then retry"). See [Switch Transitions](#switch-transitions) for the full rationale and validation rules.

**Tasks that fire external side effects must be idempotent.** `flow.Retry`, foreman lease-recovery after a worker dies mid-step, and Subgraph re-entry all replay the task body from the top. A task that charges a card, sends an email, or writes to a non-transactional store must either carry its own dedupe key (per-flow idempotency id, per-step natural key) or check first whether the effect has already happened. The framework does not deduplicate side effects for you. Tasks that are pure computation over state need no special treatment - replay just recomputes the same output.

## Time Budgets and Long-Running Work

Each task step has an execution time budget enforced by the foreman as an HTTP call timeout. The foreman's `TimeBudget` config is the hard ceiling applied to every task dispatch: it defaults to `2m`, and its validation rejects anything outside `1s`-`15m`. A task endpoint declares its own, shorter budget on its subscription with the framework option `sub.TimeBudget(duration)`; the connector then shortens that task handler's context deadline to the smaller of the caller-provided budget and the declared one, so the task is cancelled at its declared bound. The budget is a property of the task endpoint, not of the workflow graph - the graph carries no timing.

If a task exceeds its budget, the foreman cancels the HTTP call and the step's lease eventually expires for `pollPendingSteps` to recover. A task that intrinsically needs more than 15 minutes to finish must not block the worker - instead, use one of the two patterns below to free the worker between work and completion.

### Pattern 1: Interrupt and External Resume

Use when some external signal will indicate completion (a callback from an async service, a webhook arrival, human approval). The task triggers the long-running operation, then calls `flow.Interrupt(payload)` and returns. The worker is freed immediately and the flow parks in `interrupted` status. Some other process (an inbound event sink, a webhook handler, a UI action) calls `foremanapi.Resume(flowID, data)` when the operation completes; the task re-runs with the resume data merged into state.

```go
func (svc *Service) WaitForApproval(ctx context.Context, flow *workflow.Flow, requestID string, approved bool) (approvedOut bool, err error) {
    if approved {
        // Re-entry: Resume was called with approval data merged into state.
        return true, nil
    }
    // First run: kick off the approval request, then park.
    if err := svc.sendApprovalRequest(ctx, requestID); err != nil {
        return false, errors.Trace(err)
    }
    flow.Interrupt(map[string]any{"requestID": requestID}, nil)
    return false, nil
}
```

This pattern assumes another process exists that knows when to call `Resume`. Wait time is unbounded - could be seconds or weeks.

### Pattern 2: Polling with Sleep and Retry

Use when no external trigger exists and the task must check for completion itself (waiting on a job ID, a batch import, a slow third-party API). The polling task checks completion; if not done, it calls `flow.Retry` (with backoff) and returns; if done, it returns the result. Each polling cycle is a fresh task execution, so the worker is freed between checks rather than blocking inside `time.Sleep`.

```go
func (svc *Service) PollJobStatus(ctx context.Context, flow *workflow.Flow, jobID string) (jobResult string, err error) {
    status, result, err := svc.checkJobStatus(ctx, jobID)
    if err != nil {
        return "", errors.Trace(err)
    }
    switch status {
    case "running":
        // Not done yet - schedule another poll. The give-up horizon bounds total wait.
        if !flow.Retry(30*time.Second, 1.0, 0, time.Hour) { // poll every 30s, give up after 1h
            return "", errors.New("job did not complete within poll budget")
        }
        return "", nil // Retry returned true; foreman will re-run this task after the delay.
    case "failed":
        return "", errors.New("job failed")
    default:
        return result, nil
    }
}
```

A separate "kick off the job" task should run before the polling task in the graph, writing `jobID` into state for the poller to read. The two patterns can be combined: trigger work in one task, then either interrupt-and-resume or poll in a downstream task to await completion.

## Running Workflows

**Boundary rule.** A workflow microservice exposes its graph(s) and tasks, nothing else. Do not wrap the foreman in pass-through endpoints like `RunMyWorkflow`, `StartMyWorkflow`, `CancelMyWorkflow`, `MyWorkflowStatus`, etc. - they add a hop, hide foreman options (`FlowOptions`, notify-hostname) behind a narrower surface, and pull the foreman into the microservice's call graph for no benefit. Two consequences follow:

- **To launch a workflow, the upstream caller talks to the foreman directly** via `foremanapi.NewClient(svc).Run(...)` (or `Create` + `Await`; `Create` auto-runs). The workflow's microservice is not in the launch path.
- **To call another unit of work from inside a task body, use a subgraph** - `otherapi.NewSubgraph(flow).OtherWorkflow(ctx, args...)` (embeds a whole workflow as a child flow), or the raw `flow.Subgraph` for dynamic dispatch. A bare task is not independently invocable from a task body; to run one unit of work in isolation, declare a one-node workflow and subgraph it. Do **not** call `foremanapi.NewClient(svc).Run`, nor the **test-only** `Executor`, from a task body: that spawns a separate flow (or a foreman-coupled call) and forfeits re-entry, single-flow audit trail, cascading cancellation, and interrupt propagation. (The `Executor` carries a `service.Publisher` and exists only to drive endpoints from tests.)

The Foreman core service orchestrates workflow execution. Use `foremanapi.NewClient(svc)` to interact with it:

```go
import "github.com/microbus-io/fabric/coreservices/foreman/foremanapi"

// Run a workflow synchronously (create-and-run + await). Returns a *workflow.FlowOutcome with
// Status, State, plus the populated side-channel for terminal/parked statuses (Error when
// Status="failed", InterruptPayload when Status="interrupted", CancelReason when Status="cancelled").
// The Go-level err is for transport/foreman/timeout failures only; a workflow that fails surfaces
// as outcome.Status == "failed" with outcome.Error populated.
outcome, err := foremanapi.NewClient(svc).Run(ctx, myworkflowapi.MyWorkflow.URL(), initialState, nil)

// Fine-grained control: Create auto-runs and returns the running flow's key; there is no separate Start.
flowID, err := foremanapi.NewClient(svc).Create(ctx, myworkflowapi.MyWorkflow.URL(), initialState, nil)
outcome, err = foremanapi.NewClient(svc).Await(ctx, flowID)

// Inspect a running flow
outcome, err := foremanapi.NewClient(svc).Snapshot(ctx, flowID)

// Cancel a flow; the reason is recorded on every flow in the chain as outcome.CancelReason.
err = foremanapi.NewClient(svc).Cancel(ctx, flowID, "user requested")

// Resume an interrupted flow with new data
err = foremanapi.NewClient(svc).Resume(ctx, flowID, resumeData)

// Continue a thread with new input (multi-turn pattern). Returns a running flow.
// The threadKey can be any flowKey belonging to the thread; Continue finds the latest completed flow and
// inherits the thread's policy. It takes no options - use Create with Opts.ThreadKey to set policy explicitly.
newFlowID, err := foremanapi.NewClient(svc).Continue(ctx, threadKey, additionalState)
```

For typed workflow invocation in tests, use the generated `Executor` with a `WorkflowRunner`:

```go
foremanClient := foremanapi.NewClient(tester)
exec := myworkflowapi.NewExecutor(tester).WithWorkflowRunner(foremanClient)
output1, status, err := exec.MyWorkflow(ctx, input1, input2)
```

The `WorkflowRunner` interface is defined in the API package to avoid a `foremanapi` dependency. `foremanapi.Client` satisfies it implicitly.

### Flow Options (priority, fairness, delayed start)

A flow can carry scheduling hints via `*workflow.FlowOptions` (a flow-level concept, so it lives in the `workflow` package, not `foremanapi`):

```go
opts := &workflow.FlowOptions{
    Priority:       1,        // explicit priority is >= 1, lower runs first; 0 = unset -> the foreman DefaultPriority config
    FairnessKey:    "tenant-42", // "" -> derived from the tid/tenant actor claim, else the "" bucket
    FairnessWeight: 2.0,      // relative dispatch share of the fairness key; 0 -> 1
    ThreadKey:      "",       // a flowKey in an existing thread to join; "" starts a fresh thread
}
outcome, err := foremanapi.NewClient(svc).Run(ctx, myworkflowapi.MyWorkflow.URL(), initialState, opts)
flowID, err := foremanapi.NewClient(svc).Create(ctx, myworkflowapi.MyWorkflow.URL(), initialState, opts)
```

`FlowOptions` authors a flow's policy **once at genesis** (`Create`/`Run`). Derived operations inherit it and take no options: `Continue` inherits the **thread's** policy, `Fork` inherits the **origin's**, and subgraphs inherit the **parent's** - choosing a derived operation *is* choosing "inherit." `Create` with `Opts.ThreadKey` is the explicit-policy escape hatch: join an existing thread while setting policy yourself. The foreman owns all defaulting; pass `nil` (or leave a field zero) to take the default. The `DefaultPriority` config on `foreman.core` sets the priority assigned when the caller specifies none.

There is no time-delayed start option. For a deferred start, give the workflow an entry task that calls `flow.Interrupt`; the flow rests as `interrupted` until a `Resume` releases it (the event trigger). `flow.Sleep` inside a task is the in-flow wait.

The generated per-workflow `Executor` methods keep their minimal signatures. To run a *typed* workflow with options in a test, attach them once with the `WithFlowOptions` builder:

```go
exec := myworkflowapi.NewExecutor(tester).
    WithWorkflowRunner(foremanClient).
    WithFlowOptions(&workflow.FlowOptions{Priority: 1})
output1, status, err := exec.MyWorkflow(ctx, input1, input2) // unchanged signature
```

Scheduling semantics (active): the foreman dispatches by a two-level selection over the pending steps.

- Priority is a strict band. The lowest priority number with due pending work runs first, with no aging and no
  safety valve: a continuously fed stream of higher-priority flows will starve a lower-priority flow for as
  long as the stream lasts. This starvation is by design - use priority only for genuine urgency tiers, not as
  a soft hint.
- Within the chosen priority band, a fairness key is picked by weighted random (share proportional to
  `FairnessWeight`), then that key's oldest pending step runs. The pick is re-rolled per dispatch, so expected
  throughput per key tracks its weight regardless of how deep each key has backlogged - a heavy key cannot
  monopolize a light one. Order within a single key is strict FIFO by creation.
- The weight used for a key is taken from that key's oldest pending step. Supplying inconsistent
  `FairnessWeight` values for the same key across flows is the caller's responsibility (the behavior is
  well-defined but the effective weight is then whichever flow is currently oldest).

Relevant `foreman.core` configs: `DefaultPriority` (priority when the caller specifies none), `Workers` (the
per-replica candidate-cache size and dispatch parallelism), and `TimeBudget` (the hard per-step dispatch
ceiling; a task may declare a shorter `sub.TimeBudget` of its own).

## Flow Stop Notifications

To receive an event when a flow stops (completed, failed, cancelled, or interrupted), opt in with
`FlowOptions.NotifyOnStop` at `Create` (the foreman records the caller's host from the request frame, so no
hostname is passed), then subscribe to the `OnFlowStopped` event:

```go
// Opt in at Create (which auto-runs).
flowID, err := foremanapi.NewClient(svc).Create(ctx, myworkflowapi.MyWorkflow.URL(), initialState,
	&workflow.FlowOptions{NotifyOnStop: true})

// Subscribe to notifications
foremanapi.NewHook(svc).ForHost(svc.Hostname()).OnFlowStopped(
	func(ctx context.Context, flowKey string, outcome *workflow.FlowOutcome) error {
		// Handle flow stop: outcome.Status / outcome.State / outcome.Error / outcome.InterruptPayload.
		// flowKey identifies which flow stopped (it is not part of the outcome).
		return nil
	},
)
```

## Debugging Workflows

- `foremanapi.NewClient(svc).History(ctx, flowID)` - step-by-step execution trace with state snapshots and nested subgraph history
- `foremanapi.NewClient(svc).Fork(ctx, stepKey, stateOverrides)` - clone a terminal flow's prefix up to the chosen step into a new self-contained flow and re-run from there with modified state, inheriting the origin's scheduling/baggage. The original flow is immutable; Fork never mutates it. A terminal flow is recovered only by forking, not by an in-place restart.

## Mocking Workflows

Workflows can be mocked at two levels of fidelity. Both require the **real foreman** in the test app.

**Tier 1: Workflow-level mock (common case)** - Collapses the entire workflow into a single synthetic task. The handler receives the `*workflow.Flow` carrier with access to state and control signals:

```go
creditflowMock := creditflow.NewMock()
creditflowMock.MockCreditApproval(func(ctx context.Context, flow *workflow.Flow, applicant creditflowapi.Applicant, faultInjection string) (approved bool, creditVerified bool, employmentFailures int, identityVerified bool, err error) {
	return true, true, 0, true, nil
})

app := application.New()
app.Add(svc, foreman.NewService(), creditflowMock, tester)
app.RunInTest(t)
```

Under the hood, `MockCreditApproval` overrides the graph endpoint to return a trivial single-task graph and subscribes a synthetic task handler on the mock. Pass `nil` to clear the mock.

**Tier 2: Individual task mocks (rare case)** - For fine-grained control over specific tasks (interrupt/resume, per-step assertions), mock individual task endpoints while letting the real graph drive execution:

```go
creditflowMock := creditflow.NewMock()
creditflowMock.MockVerifyCredit(func(ctx context.Context, flow *workflow.Flow, creditScore int, faultInjection string) (creditVerified bool, err error) {
	return true, nil
})
```
